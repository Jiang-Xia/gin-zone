# IM（H5）后台管理系统改造方案（gin-zone）

> 目标：在现有 `gin-zone` 的“聊天/群组/好友关系/聊天记录 + admin 管理端雏形”基础上，扩展为**可运营、可审计、可风控**的后台管理系统。  
> 约束：遵守项目守护规则——分层不越界、鉴权不缺失、身份不信任前端、响应结构不漂移、契约变更必联动、错误不可吞、敏感信息不可落日志。

## 1. 现状盘点（已具备能力）

### 1.1 后端（server）
- **管理端聊天模块雏形**：`server/app/controller/admin/chat.go`
  - 好友/群聊关系列表与删除（`z_chat_friends`）
  - 群组列表（`z_chat_group` join `z_user`）
  - 管理端更新群信息、移除群成员（含“群主/管理员”权限校验）
- **聊天服务层**：`server/app/service/chat.go`
  - 群、群成员、聊天记录等基础能力

### 1.2 管理端（admin）
- API：`admin/src/api/modules/chatAdmin.ts`
- 页面：`admin/src/pages/chat/*`（friends、groups、manage）

## 2. 总体目标与设计原则

### 2.1 目标
- **可管控**：用户/群组/消息可查可处置（封禁、禁言、撤回、解散、转移）
- **可追溯**：所有管理端写操作可审计（含操作者、对象、理由、IP、时间、前后值）
- **可扩展**：后续可平滑接入内容审核、推送、运营位、数据分析、告警系统

### 2.2 原则（落地约束）
- **分层不越界**：controller 只做鉴权/参数校验/响应；复杂编排在 service；service 不依赖 Gin Context
- **鉴权与权限**：需要登录的管理端接口必须挂 JWT；资源操作必须校验权限（RBAC）
- **身份不信任前端**：业务身份以 token 解析出的用户为准；禁止用前端传入 userId 作为权限依据
- **响应结构不漂移**：统一使用 `response.Success/Fail`
- **契约变更必联动**：server 接口变更需同步检查：
  - `admin/src/api/modules/*`
  - `client/common/request/apis/*`

### 阶段 B（风控与合规，建议 2–4 周）
目标：完善“关系链、举报、导出、禁言禁图、媒体审核队列”，支撑合规审计与风险处置。

#### B1. 好友与关系链
- 黑名单管理（拉黑/解除、强校验拦截）
- 好友申请管理（管理员代处理）
- 关系解绑与数据口径清晰化（建议中期拆分：好友关系 vs 会话列表）

#### B2. 用户管控增强
- 禁言、禁发图片（支持到期时间、原因、操作者、可批量）
- 强制下线（token 版本号或黑名单策略）

#### B3. 消息合规与导出
- 导出任务：CSV/JSON（异步任务表 + 下载）
- 媒体审核队列：图片/文件消息进入待审队列（先人工，后接模型）

#### B4. 投诉举报
- 举报单流转（待处理/已处理/驳回）+ 证据留存
- 处理动作联动（封禁/撤回/禁言等）

### 阶段 C（运营与数据化，建议 4–8 周）
目标：仪表盘、运营位、推送、深度统计分析与告警体系。

#### C2. 公告与运营位
- 全站公告（定时发布、渠道：H5/APP）
- 运营位：开屏/Banner/入口配置（投放时间、排序、跳转）

#### C3. 推送配置与模拟
- 推送通道配置（Key/开关）

#### C4. 数据分析报表
- 留存、流失、群活跃排行、消息类型分布、媒体占用

## 4. 数据模型（增量表结构方案）

> 原则：**尽量增量扩展**，避免推翻现有表；中期再做“关系表/会话表拆分”。

### 4.1 阶段 A 必选新增表
- `z_admin_user` / `z_admin_role` / `z_admin_perm` / `z_admin_role_perm`：RBAC
- `z_admin_audit_log`：管理员操作审计日志
- `z_sys_config`：系统配置（注册开关、文件限制、陌生人私聊等）
- `z_system_message`：系统消息投递（可扩展为公告/推送）
- `z_sensitive_word` / `z_sensitive_hit_log`：敏感词与命中日志

### 4.2 阶段 B 推荐新增表
- `z_user_restrict`：封禁/禁言/禁图（含到期、原因、操作者）
- `z_user_block`：黑名单
- `z_friend_apply`：好友申请
- `z_export_task`：导出任务（消息导出/审计导出）
- `z_report`：举报单
- `z_media_audit_task`（可选）：媒体审核队列

### 4.3 阶段 C 推荐新增表
- `z_stat_daily_user` / `z_stat_daily_msg` / `z_stat_daily_group`：按日聚合统计

### 4.4 关键索引建议（避免检索/导出慢）
- `z_chat_log` 建议补齐组合索引（按实际查询口径选择）：
  - `(created_at)`
  - `(sender_id, created_at)`
  - `(group_id, created_at)`
  - `(msg_type, created_at)`

## 5. API 方案（按模块拆分，契约不漂移）

### 5.1 统一约定
- 管理端接口统一前缀：`/admin/*`
- 必须 JWT 鉴权；并进行 RBAC 权限点校验
- 统一响应：`response.Success/Fail`

### 5.2 阶段 A API（建议清单）
> 说明：以下为“建议接口面”，具体路径/参数可按现有风格微调，但不要让页面直写 URL。

#### 用户
- `GET /admin/users`：用户列表（分页、筛选）
- `GET /admin/users/{uid}`：用户详情
- `POST /admin/users/{uid}/restrict`：封禁/解封（先做限制登录）
- `POST /admin/users/{uid}/systemMessage`：发送系统消息

#### 群组
- `GET /admin/chat/groups`：群组列表（已有）
- `GET /admin/chat/groups/{id}`：群组详情（建议新增）
- `DELETE /admin/chat/groups/{id}`：解散群（建议新增，注意事务）
- `POST /admin/chat/groups/{id}/transferOwner`：转移群主（建议新增）
- `DELETE /admin/chat/groupMembers/{groupId}/{userId}`：踢人（已有）

#### 消息
- `GET /admin/chat/messages`：消息检索（建议新增）
- `POST /admin/chat/messages/revoke`：撤回消息（建议新增，软撤回）
- `POST /admin/chat/messages/delete`：删除消息（建议新增，软删除）

#### 敏感词
- `GET /admin/sensitiveWords`
- `POST /admin/sensitiveWords`
- `PUT /admin/sensitiveWords/{id}`
- `DELETE /admin/sensitiveWords/{id}`
- `GET /admin/sensitiveWords/hits`：命中记录（建议新增）

#### 配置与审计
- `GET /admin/sysConfig` / `POST /admin/sysConfig` / `PATCH /admin/sysConfig/{id}` / `DELETE /admin/sysConfig/{id}`：系统配置
- `GET /admin/auditLogs`：操作日志查询

## 6. Admin 前端落地（页面与复用约束）

### 6.1 复用约束（避免重复封装）
- API 必须放在 `admin/src/api/modules/*`
- token 注入/错误归一化由 `admin/src/api/http.ts` 处理
- 页面提示优先复用 `useApiMessage`
- 标准列表页优先复用 `useListPage`
- 弹窗表单优先复用 `useDialogForm`

### 6.2 阶段 A 页面清单
- 用户列表 / 用户详情（封禁、发系统消息）
- 群组列表 / 群组详情（成员分页、转移群主、解散）
- 消息检索页（筛选 + 表格 + 撤回/删除）
- 敏感词管理（增删改查）
- 系统配置（KV 表单）
- 操作日志（按操作者/时间/模块筛选）

## 7. 风险与取舍（提前约定口径，减少返工）

### 7.1 “好友关系表 vs 会话表”
- 当前 `z_chat_friends` 混合了“关系/会话/群会话”。短期可继续用（避免大改）。
- 中期建议拆出 `z_friend_relation`（双向关系）以避免权限与统计口径混乱；`z_chat_friends` 保留做会话与 last_read_time 等。

### 7.2 消息删除策略
- 建议优先 **软删除/撤回标记**，保留审计链路。
- 硬删仅在明确合规要求下执行，并确保导出/审计仍可追溯。

### 7.3 在线/活跃口径
- MVP 先用“最后登录/最后发言”近似。
- 阶段 C 再接入心跳/在线大盘（Redis 在线集合 + 定时落库）。

## 8. 实施顺序（建议）
- 第 1 周：RBAC + 审计日志 + 用户列表/详情（含封禁、系统消息）
- 第 2 周：群组详情/解散/转移 + 消息检索/撤回 + 敏感词 + 配置页
- 第 3–6 周：黑名单/好友申请/举报/导出/禁言禁图/媒体审核队列
- 第 7–12 周：仪表盘、统计聚合、公告运营位、推送模拟、告警
